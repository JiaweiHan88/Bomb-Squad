# Story 8.4: Server-Authoritative Timer & Strike Escalation

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a team,
I want the countdown owned by the server with strike-based acceleration,
so that timing is fair, tamper-proof, and pressure escalates with mistakes.

## Acceptance Criteria

1. **Given** a round starts, **When** the timer begins, **Then** the server is the sole authority on expiry and broadcasts a `TimerState` descriptor only on change (start/strike/pause/resume). (FR34, AR6/AR7, ADR-005)
2. **Given** a strike is recorded, **When** the timer rebases, **Then** the `speedMultiplier` compounds by the configured % (default 25% â†’ Ă—1.00/Ă—1.25/Ă—1.56) applied from that moment. (FR33)
3. **Given** the server-authoritative clock reaches expiry, **When** the deadline passes, **Then** the server declares the round failed â€” never a client's local clock. (FR34)

## Dependencies & Scope Boundaries (READ FIRST)

This story sits in the **middle** of the game-loop dependency graph: it is built **after** 8.3 (round start) but **ahead of** 8.2 (bomb generation), the server-side module-interaction handler, 8.5 (round resolution), and 8.7 (pause). The fences below are deliberate decisions, not oversights. **Read this whole section before writing a line of code â€” the single biggest risk in this story is building into 8.5's or 8.7's territory.**

- **This is a SERVER-ONLY story for the timer.** The visible diegetic timer LCD with client extrapolation is **Story 4.4 (backlog)** and the strike indicator/roll-up HUD is **Story 4.5 (backlog)**. The client already has everything 8.4 needs to talk to: `useGameStore` exposes `setTimer`/`setStrike`, and `bindServerEvents.ts` already wires `TIMER_UPDATE â†’ setTimer` and `STRIKE â†’ setStrike`. **Do NOT build client extrapolation, a countdown display, or a strike HUD.** 8.4 makes the server emit correct `TimerState`/`STRIKE`; 4.4/4.5 render them. The only acceptable client touch is *none* â€” verify the existing store/bind wiring is sufficient and stop. (If a trivial display is unavoidable for your own manual verification, gate it behind a dev-only seam and call it out â€” but the AC does not require it.)

- **Story 8.2 (bomb generation) is backlog â€” there is NO bomb in Redis at round start.** 8.3 left the seam `// Story 8.2: per-team bomb generation slots in here` in `ROUND_START`. The timer therefore mints and runs **independently of any bomb**. Your timer-mint seam (`// Story 8.4: TimerState is minted and broadcast here`, already present at `sessionHandlers.ts:703`) runs whether or not 8.2 has landed. Do **not** make timer minting depend on a `BombState` existing.

- **The server-side MODULE_INTERACT handler does NOT exist yet â€” strikes cannot be produced by gameplay in this worktree.** `MODULE_INTERACT` is typed in `ClientToServerEvents` but unhandled server-side; the pure `bombReducer` (which rolls a transient `'struck'` module status up into `BombState.strikes`) exists and is tested, but nothing calls it over a socket, and there is no bomb to call it against (see above). **Consequence:** AC-2 ("Given a strike is recorded") cannot be exercised end-to-end through a real module click in this story. 8.4 owns the **timer's response** to a recorded strike, not the recording. Build the strikeâ†’timer coupling as (a) a pure function and (b) a named server entry point, and **test it by driving that entry point directly** in the integration harness. The wiring that calls it from a real `MODULE_INTERACT` over a generated bomb is owned by 8.2 / the Epic-5 server-interaction story. Leave a clearly-labelled seam; do **not** stub a fake/dev "press to strike" path into production code.

- **Story 8.5 (Round Resolution) owns the round-end CEREMONY â€” 8.4 owns only the authoritative EXPIRY DECISION.** AC-3 requires the server (not a client) to *declare* the round failed at expiry. That declaration is 8.4's: the server detects expiry against its own clock and broadcasts the authoritative timeout. **What 8.4 does NOT do** (all 8.5): record `elapsedMs` into `TeamState.cumulativeTimeMs`, flip `SessionState.status` to `'between-rounds'`, play the explosion scene / 3-second hold, emit `SCOREBOARD`, or write anything to Postgres. 8.4's expiry firing broadcasts the minimal authoritative failure signal (`BOMB_EXPLODED` to the team room â€” the event already exists; populate `{ teamId, elapsedMs }` with the wall-clock elapsed at the moment of expiry) and freezes/clears that team's timer. Leave the seam `// Story 8.5: round-resolution ceremony (record time, status flip, scene, scoreboard) hooks here`. **The third-strike explosion is likewise 8.5, not 8.4** (see strike-escalation decision below).

- **Story 8.7 (Pause) owns the pause/resume TRIGGERS â€” 8.4 builds pause/resume only in the pure engine.** `TimerState` already models `pausedAt`, and the segment-reset contract in `packages/shared/src/types/timer.ts` already specifies pause/resume semantics. 8.4 implements `pause`/`resume` as pure functions (the engine is incomplete and the expiry scheduler is **unsafe** without pause-awareness) and makes the expiry scheduler **tolerate a paused timer** (a reload-and-revalidate scheduler computes `isExpired` using `pausedAt`, so a paused clock is never expired â†’ the scheduler no-ops rather than wrongly exploding a paused bomb). 8.4 does **NOT** add the `FACILITATOR_PAUSE`/`FACILITATOR_RESUME` socket handlers, disconnect-auto-pause, or the resume-reschedule wiring â€” those are 8.7. AC-1's "broadcasts on change (â€¦ pause/resume)" is satisfied at the engine + broadcast-helper level; the pause/resume *events* land in 8.7.

- **Story 8.1 (`ROUND_CONFIGURE`) is backlog.** The timer duration and strike speed-up come from the session's existing `config` (`RoundConfig.timerMs`, `RoundConfig.strikeSpeedUpPct`), populated with `DEFAULT_ROUND_CONFIG` (`timerMs: 300_000`, `strikeSpeedUpPct: 25`) at `SESSION_CREATE`. Read them off `SessionState.config` as-is; do not implement `ROUND_CONFIGURE`.

- **Single live team for now.** Sequential-relay sequencing (which team's bomb is "live") is **Story 8.9**. 8.3's `ROUND_START` assigns defusers for *all populated teams* and joins each team's sockets to `teamRoom(sessionId, teamId)`. 8.4 mints a timer **per populated team** and broadcasts to each team room. This is forward-compatible with 8.9 and costs nothing now.

- **Pure-reducer discipline (Pattern 2 / ADR):** `timerCore.ts` imports nothing from `socket.io`/`ioredis`/`pg`/`fastify`; no `Date.now()`/`Math.random()`/`setTimeout`; `now` is always an argument; invalid input returns a clamped/same-reference value, never throws. The architecture explicitly states (line 202): "No `Date.now()`, no `Math.random()`, no `setTimeout` inside a reducer â€” time and randomness are inputs passed in via state/action."
- **Timer Authority (Pattern 5 / ADR-005):** server broadcasts a `TimerState` descriptor **only on change** (start/strike/pause/resume) â€” never per tick. The client extrapolates per frame (that's 4.4). **Expiry authority lives on the server** â€” "A bomb explodes when the **server** says so, never when a client's local clock hits zero." The scheduler is the embodiment of "the server schedules its own authoritative expiry check."
- **Strike escalation (Pattern 5):** changes `speedMultiplier` and **rebases** `startedAt`/`remainingAtStart` so the new rate applies from that moment (compounding Ă—1.00 â†’ Ă—1.25 â†’ Ă—1.56).
- **Rooms (Pattern 1):** timer/strike/explosion broadcasts are **team-scoped** â†’ `teamRoom(sessionId, teamId)`, not the whole session. 8.3 already joins sockets to team rooms at `ROUND_START`.
- **Server owns all I/O (Pattern 2):** the scheduler and handlers read the clock and touch Redis/sockets; the pure core never does. Persist-then-emit; never fire-and-forget an await.
- **Redis keyspace:** `timerKey` only (O(1)); no scans. `session:{id}:team:{teamId}:timer â†’ TimerState` is the documented mapping (architecture line 407).
- **Logging:** structured, keyed by `sessionId`/`teamId`/`roundNumber`; strikes/round-results at info (architecture line 385); never log join codes (AR15) or tokens.

### Design decisions settled for this story (do not re-litigate, do not silently change)

1. **The clock is injected, never read inside pure code.** One `clock()` (default `Date.now`) lives in the scheduler/handler layer; `now` is read once per handler invocation and threaded into every pure call. Tests inject a fake clock. This is the only way to make timer logic deterministically testable and is the project's stated rule.
2. **The expiry scheduler reloads + revalidates on every fire.** It never explodes off the in-memory timer the wake was scheduled from. This single discipline makes the scheduler robust to strikes (deadline moved), pause (not expired), and already-resolved rounds (key gone) **without** any of those stories being built yet.
3. **A paused timer is structurally non-expirable.** `isExpired`/`remainingMs` substitute `pausedAt` for `now`, so the scheduler firing during a (future 8.7) pause is a safe no-op. 8.4 builds the pure pause/resume but not the handlers.
4. **STRIKE carries the rebased timer; no duplicate TIMER_UPDATE on strike.** `StrikePayload.timer` is the authoritative timer for a strike-driven rebase. (Resolves the deferred-work redundancy note.) `TIMER_UPDATE` is for non-strike changes (round start; later, pause/resume in 8.7).
5. **The 3rd strike does not escalate the timer â€” it ends the round (8.5).** `escalateOnStrike` early-returns at `strikes >= 3`. The timer's multiplier domain is therefore {Ă—1.00, Ă—1.25, Ă—1.56} at default. (Resolves the `StrikeCount`-resting-at-3 deferral: the type may rest at 3, but no timer math ever runs on a 3rd strike.)
6. **Expiry recording is minimal â€” the ceremony is 8.5.** 8.4 broadcasts `BOMB_EXPLODED { teamId, elapsedMs }` and clears the team timer; it does NOT write `cumulativeTimeMs`, flip `status`, animate, or persist to Postgres. `elapsedMs` at timeout = `config.timerMs` (the displayed clock, by definition, reached 0).
7. **Per-populated-team timers.** One `TimerState` per team that received a defuser at `ROUND_START`, each broadcast to its own team room. Forward-compatible with 8.9's relay sequencing; no "live team" gating yet.
8. **Single-process, in-memory scheduler (V1).** Not durable across restart, not multi-instance. Accepted posture, commented; no Redis-backed scheduling.

### Pitfalls from prior reviews (this codebase's known sharp edges)

- **`socket.id` is player identity (accepted V1):** a mid-round refresh orphans the player and their team membership. Do not introduce any new dependence on `socket.data` for authority; the timer is keyed by `sessionId:teamId`, not by socket, so it is unaffected by a single player's refresh.
- **Transient-`'struck'` rendering (deferred-work, owned here):** the client LED reverts a `'struck'` module to the armed visual after a 600 ms flash; a module *resting* at `'struck'` is visually indistinguishable from armed. 8.4 does not broadcast module state, but its `STRIKE` event is the bomb-level signal the strike HUD (4.5) keys off. Make the `STRIKE` contract explicit in code comments: it carries the **new absolute** team strike total (not a delta) and the rebased timer â€” 4.5 will render from it. Do not write anything that precludes the transient-struck module broadcast the future interaction handler will pair with `STRIKE`.
- **Non-atomic multi-key writes (accepted):** `ROUND_START` already does a non-atomic two-key write; adding per-team timer writes keeps the same posture. Do not add locks/WATCH; extend the existing comment.
- **`fetchSockets()` is async** â€” already awaited in the existing team-join loop; your timer broadcast runs after it, so team rooms are populated.
- **Scheduler/shutdown ordering:** dispose the scheduler **before** `io.close()` so a wake can't fire `io.to(...).emit` into a closing server. The existing `shutdown()` is the place.
- **Floating-point multiplier:** Ă—1.25Â˛ = 1.5625 exactly in IEEE-754 (0.25 and its products are representable), so assert the exact value in tests; but do not assume all `strikeSpeedUpPct` values are exact â€” use a tolerance for non-power-of-two percentages (e.g. 30% â†’ 1.69).

# Story 8.4: Server-Authoritative Timer & Strike Escalation

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a team,
I want the countdown owned by the server with strike-based acceleration,
so that timing is fair, tamper-proof, and pressure escalates with mistakes.

## Acceptance Criteria

1. **Given** a round starts, **When** the timer begins, **Then** the server is the sole authority on expiry and broadcasts a `TimerState` descriptor only on change (start/strike/pause/resume). (FR34, AR6/AR7, ADR-005)
2. **Given** a strike is recorded, **When** the timer rebases, **Then** the `speedMultiplier` compounds by the configured % (default 25% → ×1.00/×1.25/×1.56) applied from that moment. (FR33)
3. **Given** the server-authoritative clock reaches expiry, **When** the deadline passes, **Then** the server declares the round failed — never a client's local clock. (FR34)

## Dependencies & Scope Boundaries (READ FIRST)

This story sits in the **middle** of the game-loop dependency graph: it is built **after** 8.3 (round start) but **ahead of** 8.2 (bomb generation), the server-side module-interaction handler, 8.5 (round resolution), and 8.7 (pause). The fences below are deliberate decisions, not oversights. **Read this whole section before writing a line of code — the single biggest risk in this story is building into 8.5's or 8.7's territory.**

- **This is a SERVER-FIRST story; the client display already EXISTS — do not rebuild it.** The visible diegetic timer LCD with client extrapolation (**Story 4.4**) and the strike indicator/roll-up HUD (**Story 4.5**) are **DONE on master and now merged into this worktree** (merge `8f3db65`, 2026-06-13). The full client receive-side is already wired: `useGameStore.setTimer`/`setStrike`; `bindServerEvents.ts` routes `TIMER_UPDATE → noteTimerBroadcast(timer) → setTimer` and `STRIKE → setStrike`; `apps/client/src/scenes/TimerLcd.tsx` (+ pure `timerLcd.ts`) extrapolates the countdown per frame via `serverNow()`; `apps/client/src/scenes/StrikeIndicator.tsx` (+ pure `strikeIndicator.ts`) renders `bomb.strikes`; both are mounted inside `BombScene` (which 8.3's `ActiveRound` already shows the defuser). **Do NOT re-implement extrapolation, a countdown, or a strike HUD — they are built and tested.** 8.4's job is to make the server emit correct `TimerState`/`STRIKE`/`BOMB_EXPLODED`; those existing components then render live. **This makes 8.4's verification surface real** (a visible LCD that counts down and detonates), upgraded from the "logs only" note this story originally carried.

- **HARD INTERFACE CONTRACT imposed by 4.4's `serverClock.ts` — your server emitter MUST honour it.** The client estimates `serverEpoch − clientEpoch` as `timer.startedAt − receivedAt` on **every running** `TIMER_UPDATE` (`estimateClockOffset`), and **skips paused broadcasts** (`pausedAt !== null → null`). For that estimator to stay correct, the server must guarantee: (1) **every running-segment broadcast carries a freshly-stamped server `startedAt`** — i.e. segments are always rebased on change (start/strike/resume all set `startedAt = now`), never re-broadcast with a stale `startedAt`; (2) **paused broadcasts set `pausedAt`** (so the client skips them for offset estimation). Your `timerCore` design (Task 1) already satisfies both — but this is now a *contract*, not a nicety: a running broadcast with a stale `startedAt` would corrupt the client clock and could push the displayed `0:00` **ahead** of the server (the phantom-expiry direction AC-3 forbids). `serverClock.ts` also explicitly states *"smoothing / RTT compensation is **Story 8.4's call** when the real server emitter lands"* — see the estimator decision in Dev Notes (decision: keep the single-sample estimator; the only client change 8.4 makes is the reconnect-reset below).

- **Two client cleanup items are now explicitly assigned to 8.4** (`deferred-work.md`, from the 4.4/4.5 reviews). These are the ONLY sanctioned client touches in this story; do not expand beyond them:
  - **(REQUIRED) Reset the server-clock offset on disconnect** (`apps/client/src/net/serverClock.ts` + `bindServerEvents.ts` teardown) — `clockOffsetMs` is a process-global cleared only by the test-only `resetClockOffsetForTest()`; a reconnect keeps the previous connection's offset until the first new running `TIMER_UPDATE`, momentarily biasing `serverNow()` possibly *ahead* of the server. Add a `resetClockOffset()` and call it from the `disconnect` handler (or the bind teardown). Small, well-bounded, and the file's own doc comment defers it here.
  - **(DEFER — note, do not fix) STRIKE-before-`BOMB_INIT` race** (`StrikeIndicator.tsx`, `gameStore.setStrike`) — when a `STRIKE` arrives with no `bomb` in the store, `setStrike` drops the count and the indicator floors at 0. **This stays deferred to Story 8.2**, because `BOMB_INIT` (the missing half) does not exist until 8.2 generates a bomb — in this worktree there is no bomb and no real strike pathway, so the race is unreachable. Re-confirm it in the story's deferred list; do not change `setStrike` here (a fix without `BOMB_INIT` is untestable and risks masking the real ordering work 8.2 must do).

- **Story 8.2 (bomb generation) is backlog — there is NO bomb in Redis at round start.** 8.3 left the seam `// Story 8.2: per-team bomb generation slots in here` in `ROUND_START`. The timer therefore mints and runs **independently of any bomb**. Your timer-mint seam (`// Story 8.4: TimerState is minted and broadcast here`, already present at `sessionHandlers.ts:703`) runs whether or not 8.2 has landed. Do **not** make timer minting depend on a `BombState` existing.

- **The server-side MODULE_INTERACT handler does NOT exist yet — strikes cannot be produced by gameplay in this worktree.** `MODULE_INTERACT` is typed in `ClientToServerEvents` but unhandled server-side; the pure `bombReducer` (which rolls a transient `'struck'` module status up into `BombState.strikes`) exists and is tested, but nothing calls it over a socket, and there is no bomb to call it against (see above). **Consequence:** AC-2 ("Given a strike is recorded") cannot be exercised end-to-end through a real module click in this story. 8.4 owns the **timer's response** to a recorded strike, not the recording. Build the strike→timer coupling as (a) a pure function and (b) a named server entry point, and **test it by driving that entry point directly** in the integration harness. The wiring that calls it from a real `MODULE_INTERACT` over a generated bomb is owned by 8.2 / the Epic-5 server-interaction story. Leave a clearly-labelled seam; do **not** stub a fake/dev "press to strike" path into production code.

- **Story 8.5 (Round Resolution) owns the round-end CEREMONY — 8.4 owns only the authoritative EXPIRY DECISION.** AC-3 requires the server (not a client) to *declare* the round failed at expiry. That declaration is 8.4's: the server detects expiry against its own clock and broadcasts the authoritative timeout. **What 8.4 does NOT do** (all 8.5): record `elapsedMs` into `TeamState.cumulativeTimeMs`, flip `SessionState.status` to `'between-rounds'`, play the explosion scene / 3-second hold, emit `SCOREBOARD`, or write anything to Postgres. 8.4's expiry firing broadcasts the minimal authoritative failure signal (`BOMB_EXPLODED` to the team room — the event already exists; populate `{ teamId, elapsedMs }` with the wall-clock elapsed at the moment of expiry) and freezes/clears that team's timer. Leave the seam `// Story 8.5: round-resolution ceremony (record time, status flip, scene, scoreboard) hooks here`. **The third-strike explosion is likewise 8.5, not 8.4** (see strike-escalation decision below).

- **Story 8.7 (Pause) owns the pause/resume TRIGGERS — 8.4 builds pause/resume only in the pure engine.** `TimerState` already models `pausedAt`, and the segment-reset contract in `packages/shared/src/types/timer.ts` already specifies pause/resume semantics. 8.4 implements `pause`/`resume` as pure functions (the engine is incomplete and the expiry scheduler is **unsafe** without pause-awareness) and makes the expiry scheduler **tolerate a paused timer** (a reload-and-revalidate scheduler computes `isExpired` using `pausedAt`, so a paused clock is never expired → the scheduler no-ops rather than wrongly exploding a paused bomb). 8.4 does **NOT** add the `FACILITATOR_PAUSE`/`FACILITATOR_RESUME` socket handlers, disconnect-auto-pause, or the resume-reschedule wiring — those are 8.7. AC-1's "broadcasts on change (… pause/resume)" is satisfied at the engine + broadcast-helper level; the pause/resume *events* land in 8.7.

- **Story 8.1 (`ROUND_CONFIGURE`) is backlog.** The timer duration and strike speed-up come from the session's existing `config` (`RoundConfig.timerMs`, `RoundConfig.strikeSpeedUpPct`), populated with `DEFAULT_ROUND_CONFIG` (`timerMs: 300_000`, `strikeSpeedUpPct: 25`) at `SESSION_CREATE`. Read them off `SessionState.config` as-is; do not implement `ROUND_CONFIGURE`.

- **Single live team for now.** Sequential-relay sequencing (which team's bomb is "live") is **Story 8.9**. 8.3's `ROUND_START` assigns defusers for *all populated teams* and joins each team's sockets to `teamRoom(sessionId, teamId)`. 8.4 mints a timer **per populated team** and broadcasts to each team room. This is forward-compatible with 8.9 and costs nothing now.

- [x] Task 1 — Pure timer engine in `apps/server/src/timer/timerCore.ts` (AC: 1, 2, 3)
  - [x] Create `apps/server/src/timer/timerCore.ts` — a set of **pure** functions over `TimerState` (imported from `@bomb-squad/shared`). **No `Date.now()`, no `setTimeout`, no `Math.random()`, no I/O** — `now` (epoch ms) is always a parameter. Identical discipline to `apps/server/src/session/startRound.ts` and the architecture's pure-reducer rule.
    - [x] `startSegment(durationMs: number, now: number): TimerState` → `{ startedAt: now, remainingAtStart: durationMs, speedMultiplier: 1, pausedAt: null }`. Guard: a non-finite or negative `durationMs` is clamped to `0` (defensive; the handler already validates `timerMs > 0`, but the pure fn must not produce a NaN timeline).
    - [x] `remainingMs(timer: TimerState, now: number): number` → within the current segment: `remainingAtStart - (effectiveNow - startedAt) * speedMultiplier`, where `effectiveNow = timer.pausedAt ?? now` (frozen while paused). **Clamp at 0** (never negative). This is the single source of the displayed-time formula documented in `timer.ts`.
    - [x] `isExpired(timer: TimerState, now: number): boolean` → `remainingMs(timer, now) <= 0`. A **paused** timer is never expired (because `remainingMs` substitutes `pausedAt` for `now`) — this is what makes the scheduler safe against pause without 8.7.
    - [x] `rebaseForStrike(timer: TimerState, strikeSpeedUpPct: number, now: number): TimerState` → snapshot `remainingAtStart = remainingMs(timer, now)`, `startedAt = now`, `pausedAt = timer.pausedAt` (preserve pause state — a strike while paused keeps it frozen; rebase the frozen remaining), and **compound** `speedMultiplier = timer.speedMultiplier * (1 + strikeSpeedUpPct / 100)`. Opening a fresh segment is mandatory — a single segment cannot describe a piecewise-speed timeline (see the `timer.ts` header). Clamp `strikeSpeedUpPct` into `[0, 50]` defensively (config range).
    - [x] `pause(timer: TimerState, now: number): TimerState` → if already paused, return the **same reference** (idempotent no-op); else `{ ...timer, pausedAt: now }`.
    - [x] `resume(timer: TimerState, now: number): TimerState` → if not paused, same reference; else **start a fresh segment**: `{ startedAt: now, remainingAtStart: remainingMs(timer, timer.pausedAt!), speedMultiplier: timer.speedMultiplier, pausedAt: null }`. (Simply nulling `pausedAt` would silently subtract the whole paused span — the `timer.ts` header calls this out explicitly.)
    - [x] `expiryInstant(timer: TimerState): number | null` → the absolute epoch ms at which a *running* segment hits 0 (`startedAt + remainingAtStart / speedMultiplier`); returns `null` when paused. The scheduler uses this to set its wake. (Guard `speedMultiplier > 0`.)

### Architecture compliance (non-negotiable)

- **Pure-reducer discipline (Pattern 2 / ADR):** `timerCore.ts` imports nothing from `socket.io`/`ioredis`/`pg`/`fastify`; no `Date.now()`/`Math.random()`/`setTimeout`; `now` is always an argument; invalid input returns a clamped/same-reference value, never throws. The architecture explicitly states (line 202): "No `Date.now()`, no `Math.random()`, no `setTimeout` inside a reducer — time and randomness are inputs passed in via state/action."
- **Timer Authority (Pattern 5 / ADR-005):** server broadcasts a `TimerState` descriptor **only on change** (start/strike/pause/resume) — never per tick. The client extrapolates per frame (that's 4.4). **Expiry authority lives on the server** — "A bomb explodes when the **server** says so, never when a client's local clock hits zero." The scheduler is the embodiment of "the server schedules its own authoritative expiry check."
- **Strike escalation (Pattern 5):** changes `speedMultiplier` and **rebases** `startedAt`/`remainingAtStart` so the new rate applies from that moment (compounding ×1.00 → ×1.25 → ×1.56).
- **Rooms (Pattern 1):** timer/strike/explosion broadcasts are **team-scoped** → `teamRoom(sessionId, teamId)`, not the whole session. 8.3 already joins sockets to team rooms at `ROUND_START`.
- **Server owns all I/O (Pattern 2):** the scheduler and handlers read the clock and touch Redis/sockets; the pure core never does. Persist-then-emit; never fire-and-forget an await.
- **Redis keyspace:** `timerKey` only (O(1)); no scans. `session:{id}:team:{teamId}:timer → TimerState` is the documented mapping (architecture line 407).
- **Logging:** structured, keyed by `sessionId`/`teamId`/`roundNumber`; strikes/round-results at info (architecture line 385); never log join codes (AR15) or tokens.

### Design decisions settled for this story (do not re-litigate, do not silently change)

1. **The clock is injected, never read inside pure code.** One `clock()` (default `Date.now`) lives in the scheduler/handler layer; `now` is read once per handler invocation and threaded into every pure call. Tests inject a fake clock. This is the only way to make timer logic deterministically testable and is the project's stated rule.
2. **The expiry scheduler reloads + revalidates on every fire.** It never explodes off the in-memory timer the wake was scheduled from. This single discipline makes the scheduler robust to strikes (deadline moved), pause (not expired), and already-resolved rounds (key gone) **without** any of those stories being built yet.
3. **A paused timer is structurally non-expirable.** `isExpired`/`remainingMs` substitute `pausedAt` for `now`, so the scheduler firing during a (future 8.7) pause is a safe no-op. 8.4 builds the pure pause/resume but not the handlers.
4. **STRIKE carries the rebased timer; no duplicate TIMER_UPDATE on strike.** `StrikePayload.timer` is the authoritative timer for a strike-driven rebase. (Resolves the deferred-work redundancy note.) `TIMER_UPDATE` is for non-strike changes (round start; later, pause/resume in 8.7).
5. **The 3rd strike does not escalate the timer — it ends the round (8.5).** `escalateOnStrike` early-returns at `strikes >= 3`. The timer's multiplier domain is therefore {×1.00, ×1.25, ×1.56} at default. (Resolves the `StrikeCount`-resting-at-3 deferral: the type may rest at 3, but no timer math ever runs on a 3rd strike.)
6. **Expiry recording is minimal — the ceremony is 8.5.** 8.4 broadcasts `BOMB_EXPLODED { teamId, elapsedMs }` and clears the team timer; it does NOT write `cumulativeTimeMs`, flip `status`, animate, or persist to Postgres. `elapsedMs` at timeout = `config.timerMs` (the displayed clock, by definition, reached 0).
7. **Per-populated-team timers.** One `TimerState` per team that received a defuser at `ROUND_START`, each broadcast to its own team room. Forward-compatible with 8.9's relay sequencing; no "live team" gating yet.
8. **Single-process, in-memory scheduler (V1).** Not durable across restart, not multi-instance. Accepted posture, commented; no Redis-backed scheduling.
9. **Broadcast-freshness invariant (the 4.4 `serverClock` contract).** Every *running* `TIMER_UPDATE`/`STRIKE` carries a freshly-stamped server `startedAt`; the client derives its clock offset from `startedAt − receivedAt`. Therefore segments are **always rebased on change** (start/strike/resume set `startedAt = now`) and a running timer is **never** re-broadcast with a stale `startedAt`. Paused broadcasts set `pausedAt` (the client estimator skips them). `timerCore` already enforces this; do not break it.
10. **Keep the single-sample clock estimator (V1); the only client change is the reconnect-reset.** 4.4 deferred "smoothing / RTT compensation" to 8.4. Decision: the single-sample estimator is sufficient and biases the displayed clock *behind* the server (the safe direction for AC-3); do not add RTT round-trip compensation or smoothing now. The one concrete improvement 8.4 makes is resetting the offset on disconnect (Task 5b) so a reconnect cannot carry a stale, possibly-ahead offset.

### Pitfalls from prior reviews (this codebase's known sharp edges)

- **`socket.id` is player identity (accepted V1):** a mid-round refresh orphans the player and their team membership. Do not introduce any new dependence on `socket.data` for authority; the timer is keyed by `sessionId:teamId`, not by socket, so it is unaffected by a single player's refresh.
- **Transient-`'struck'` rendering (deferred-work, owned here):** the client LED reverts a `'struck'` module to the armed visual after a 600 ms flash; a module *resting* at `'struck'` is visually indistinguishable from armed. 8.4 does not broadcast module state, but its `STRIKE` event is the bomb-level signal the strike HUD (4.5) keys off. Make the `STRIKE` contract explicit in code comments: it carries the **new absolute** team strike total (not a delta) and the rebased timer — 4.5 will render from it. Do not write anything that precludes the transient-struck module broadcast the future interaction handler will pair with `STRIKE`.
- **Non-atomic multi-key writes (accepted):** `ROUND_START` already does a non-atomic two-key write; adding per-team timer writes keeps the same posture. Do not add locks/WATCH; extend the existing comment.
- **`fetchSockets()` is async** — already awaited in the existing team-join loop; your timer broadcast runs after it, so team rooms are populated.
- **Scheduler/shutdown ordering:** dispose the scheduler **before** `io.close()` so a wake can't fire `io.to(...).emit` into a closing server. The existing `shutdown()` is the place.
- **Floating-point multiplier:** ×1.25² = 1.5625 exactly in IEEE-754 (0.25 and its products are representable), so assert the exact value in tests; but do not assume all `strikeSpeedUpPct` values are exact — use a tolerance for non-power-of-two percentages (e.g. 30% → 1.69).

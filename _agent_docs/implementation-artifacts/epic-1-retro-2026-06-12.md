## What Hurt

### The port-range incident (the big one — Jay's primary pain point)

The planned WebRTC port model (LiveKit `50000–50199/udp` + coturn `40000–40199/udp` = 400 published UDP ports) was **technically unworkable and host-degrading**: on Docker Desktop/WSL2 each published port spawns a userland proxy process; 400 publications exceeded the ~256 forwarded-port cap and overlapped the Windows-reserved `50000–50059` band. **Impact exceeded what the story records show:** the host system was wedged and required multiple restarts — failure blast radius was the *machine*, not just the compose stack.

**Provenance / root cause (why the bad number existed):**
- The original `50000–60000/udp` was **LiveKit's vendor default**, copied into the architecture at design time, then arbitrarily haircut to 200 ports — never derived from the actual topology (SFU = 1 mux port for all participants; TURN ≈ 1 relay port per peer ⇒ ~33 ports total for 16 players).

**Why it wasn't caught earlier (three filters failed the same way):**
1. Spec-fidelity culture propagated the number faithfully: architecture → project-context → epics NFR15 → Story 1.8 AC. Cross-citation *felt* like verification but was one source in five costumes.
2. Adversarial reviews audit work against the spec; they structurally cannot catch a spec that is itself wrong.
3. Nothing exercised UDP publishing until the last story of the epic (infra deliberately deferred to 1.8), so first contact with reality came at the most expensive moment.

Resolved via correct-course (sprint-change-proposal-2026-06-12): LiveKit single mux port `7882/udp` + coturn `40000–40031/udp`, all artifacts synced.

### Recurring friction

All solved and the recipes now propagate, but it was the largest source of debug-log entries.

## Key Lessons (confirmed by Jay)

1. **Externally-sourced numbers in specs need a first-principles derivation, not a citation.** Any vendor-copied figure (port ranges, limits, timeouts, sizing) gets a "derived from our topology because…" justification at architecture time, or is flagged unverified.
2. **Environment-constrained specs get a cheap spike before they're embedded in ACs.** A throwaway compose file publishing the planned range would have failed on day one of planning instead of at Story 1.8.
3. **"Verify" tasks must produce runtime evidence, not checkbox confidence.** An AC verified by inspecting the spec it came from is circular.
4. **Dev-host constraints belong in project-context.md.** WSL2 port cap, Windows-reserved UDP band, bind-mount restart quirk — agent-facing rules should carry them so they aren't rediscovered the hard way.
5. **Don't over-correct into spiking everything** (Samus's counterweight): Epic 1's infra-last sequencing was mostly right; spike the *host-facing* assumptions specifically.

## Deferred from: code review of story-1.5 (2026-06-12)

- **Signal during blocking `await redisClient.connect()` is unguarded** (`apps/server/src/index.ts:59-63,119-120`) — SIGINT/SIGTERM handlers are registered *after* the initial Redis connect, which can block for the OS TCP-connect timeout against an unreachable host. A signal in that window kills the process with no graceful close (Redis client + pg pool leak). Story 1.5 widened the pre-existing "signal before `listen()`" window (see Story 1.4 deferral above) by inserting the blocking connect. Fix alongside that item — register handlers earlier *and* gate them on listen-complete, or make the initial connect non-blocking. Deferred — low-likelihood in current scope.
- **Adapter error events bypass the pino logger and can spam on reconnect** (`apps/server/src/state/index.ts:790-792`, `apps/server/src/persistence/index.ts:558-560`) — `client.on('error')`/`pool.on('error')` use `console.error`, not `fastify.log`, so the output is unstructured and out-of-band; a permanently-down store produces an unbounded reconnect-error stream (each attempt logs). Crash is already prevented (handlers attached). Inject the Fastify logger into `connectRedis`/`connectPostgres` and consider a reconnect-log rate cap / `retryStrategy` backoff. Deferred — operational polish, non-fatal.
- **Redis key-builders don't validate `sessionId`/`teamId`** (`apps/server/src/state/keys.ts`) — raw interpolation means an ID containing `:` or an empty string silently produces a colliding/malformed key. Spec-sanctioned deferral (story Dev Notes: "keep IDs from containing `:` is a later-story concern; join codes are alphanumeric"). Revisit when ID generation/validation lands. Deferred — spec-sanctioned.

## Deferred from: code review of story-1.6 (2026-06-12)

- **No runtime guard on module-reducer output contract** (`apps/server/src/reducers/bombReducer.ts:9-14`) — `applyModuleResult` trusts the `ModuleState` a module reducer returns. Because `ModuleReducer = Reducer<ModuleState<unknown>, unknown>` erases the per-module type, a buggy reducer can return an out-of-contract `status` (anything other than `armed|solved|struck` is silently treated as not-solved/not-struck) or a different `moduleId`, permanently rebinding the slot to another (possibly unregistered) reducer. No validation, no test. Deferred — module reducers are first-party and individually typed; add a defensive guard (stable `moduleId` + legal `status`) when the Epic 5 module plugin contract is built.
